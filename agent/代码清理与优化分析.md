## 2.2 [严重] LLM 重试循环 — 85% 重复

- **重复位置 A:** `src/llm/anthropic.rs:105-149` (45 行)
- **重复位置 B:** `src/llm/openai_compat.rs:96-133` (38 行)

**相同部分:** 3 次循环 + 指数退避 + `match resp { Ok(r) if success => ..., Ok(r) if retryable => ..., Ok(r) => bail!(), Err(e) => ... }` + `Err(last_err.unwrap_or_else(...))`

**不同部分:** Anthropic 额外重试 HTTP 529；请求构建方式不同

- **建议抽象方式:**
```rust
// src/llm/mod.rs 新增
pub(crate) async fn retry_request<F, Fut>(
    max_attempts: u32,
    provider: &str,
    is_retryable: fn(u16) -> bool,
    mut make_request: F,
) -> Result<reqwest::Response>
where
    F: FnMut() -> Fut,
    Fut: std::future::Future<Output = Result<reqwest::Response, reqwest::Error>>,
{
    let mut last_err = None;
    for attempt in 0..max_attempts {
        if attempt > 0 {
            let backoff = Duration::from_millis(1000 * 2u64.pow(attempt - 1));
            tracing::warn!(attempt, ?backoff, "retrying {provider} request / 正在重试 {provider} 请求");
            tokio::time::sleep(backoff).await;
        }
        match make_request().await {
            Ok(r) if r.status().is_success() => return Ok(r),
            Ok(r) if is_retryable(r.status().as_u16()) => {
                let status = r.status();
                let text = r.text().await.unwrap_or_default();
                tracing::warn!(%status, body=%text, "retryable error from {provider}");
                last_err = Some(anyhow::anyhow!("HTTP {status}: {text}"));
            }
            Ok(r) => {
                let status = r.status();
                let text = r.text().await.unwrap_or_default();
                anyhow::bail!("{provider} non-retryable HTTP {status}: {text}");
            }
            Err(e) => { last_err = Some(e.into()); }
        }
    }
    Err(last_err.unwrap_or_else(|| anyhow::anyhow!("{provider} request failed after retries")))
}
```
调用方：
```rust
// anthropic.rs

    code == 429 || code == 529 || (500..600).contains(&code)
}

    self.http.post(ANTHROPIC_API_URL)
```

# 3. 可合并逻辑

## 3.1 [中等] `resolve_env()` 与 `resolve_env_optional()` — 90% 相同

- **当前问题:** `config/mod.rs:188-211`，两个函数共享 `strip_prefix("${").and_then(|s| s.strip_suffix('}'))` 逻辑，区别仅在于错误处理。

**优化前** (22 行):
```rust
fn resolve_env(value: &str, field_name: &str) -> Result<String> {
    if let Some(inner) = value.strip_prefix("${").and_then(|s| s.strip_suffix('}')) {
        std::env::var(inner).with_context(|| format!("...{field_name}...{inner}..."))
    } else {
        Ok(value.to_string())
    }
}

fn resolve_env_optional(value: &str) -> String {
    if value.is_empty() { return String::new(); }
    if let Some(inner) = value.strip_prefix("${").and_then(|s| s.strip_suffix('}')) {
        std::env::var(inner).unwrap_or_default()
    } else {
        value.to_string()
    }
}
```

**优化后** (14 行):
```rust
fn resolve_env_ref(value: &str) -> Option<&str> {
    value.strip_prefix("${").and_then(|s| s.strip_suffix('}'))
}

fn resolve_env(value: &str, field_name: &str) -> Result<String> {
    match resolve_env_ref(value) {
        Some(var) => std::env::var(var).with_context(|| {
            format!("Config field `{field_name}` references env var `{var}` which is not set / 配置字段 `{field_name}` 引用的环境变量 `{var}` 未设置")
        }),
        None => Ok(value.to_string()),
    }
}

fn resolve_env_optional(value: &str) -> String {
    if value.is_empty() { return String::new(); }
    resolve_env_ref(value)
        .map(|var| std::env::var(var).unwrap_or_default())
        .unwrap_or_else(|| value.to_string())
}
```

## 3.3 [低] 11 个 `impl Default for` 块（130 行纯委托）

- **当前问题:** `config/sections.rs` 中 11 个 struct 各有一个 `impl Default`，每个都是 `field: default_xxx()` 的纯委托。

- **优化方案:** 使用 `#[serde(default = "default_xxx")]` 直接标注字段，配合 struct 级 `#[serde(default)]`，可以完全省略手写 `impl Default`。

**优化前** (以 `LimitsSection` 为例, 9 行):
```rust
impl Default for LimitsSection {
    fn default() -> Self {
        Self {
            max_requests_per_minute: default_max_requests_per_minute(),
            max_message_length: default_max_message_length(),
            max_tokens_per_conversation: default_max_tokens_per_conversation(),
        }
    }
}
```

**优化后** (0 行 impl + 字段级注解):
```rust
#[derive(Debug, Clone, Deserialize, Default)]  // ← 加 Default derive
pub struct LimitsSection {
    #[serde(default = "default_max_requests_per_minute")]
    pub max_requests_per_minute: u32,
    #[serde(default = "default_max_message_length")]
    pub max_message_length: u32,
    #[serde(default = "default_max_tokens_per_conversation")]
    pub max_tokens_per_conversation: u64,
}
// 删除 impl Default for LimitsSection
```

**注意:** 此方案要求所有字段类型的 `Default::default()` 值与自定义 `default_xxx()` 返回值不一致时，必须在每个字段上加 `#[serde(default = "...")]`。对于所有字段都已标注的 struct，可以安全使用 `#[derive(Default)]` 替代手写。对于默认值非零的字段较多的 struct（如 `ToolsSection` 24 行），保留手写 `impl Default` 可能反而更清晰。**建议仅对 ≤10 行的 8 个小 struct 执行此优化，保留 ToolsSection、AgentSection、AuditSection 的手写版本。**

# 4. 性能优化点

## 4.1 [中等] `collect_workspace_extensions()` 仍在同步线程执行 I/O

- **当前实现:** `src/agent/util.rs:58-98` — 使用 `std::fs::read_dir()` 同步遍历。
- **问题:** 虽然结果已被 `skill_match.rs` 缓存 60 秒，但首次调用和缓存刷新时，同步 I/O 仍会阻塞 tokio worker 线程。在大型 workspace 中影响其他并发任务的延迟。
- **优化方案:**
```rust
// skill_match.rs 中调用处改为:
let extensions = tokio::task::spawn_blocking(move || {
    collect_workspace_extensions(&workspace_path, WORKSPACE_EXTENSION_SCAN_LIMIT)
}).await.unwrap_or_default();
```

## 4.2 [低] `SkillMeta` 的 `trigger_terms()` / `keyword_terms()` 已预计算，但 `extract_description_terms()` 每次重新计算

- **当前实现:** `skill.rs` 已将 trigger/keyword/extension 预规范化为 `normalized_*` 字段，返回 `&[String]`。但 `score_description_match()` 内部每次调用 `extract_description_terms(&description)` 仍然对 description 做 split+lowercase+stopword 过滤。
- **问题:** N 个 skills × 每条消息 = N 次重复计算相同 description 的词集。
- **优化方案:** 在 `SkillMeta` 中增加 `normalized_description_terms: Vec<String>` 字段，构造时预计算：
```rust
// skill.rs SkillMeta 构造处
normalized_description_terms: extract_description_terms(
    &description.trim().to_lowercase()
),
```

## 4.3 [低] 测试代码中大量重复的 `MockLlm + ToolRegistry + AgentCore` 初始化

- **当前实现:** `agent/mod.rs` 测试区约 10+ 个测试函数，每个都重复 ~15 行的初始化样板。
- **不是生产性能问题，但增加维护成本和编译时间。**
- **优化方案:** 提取 test fixture builder:
```rust
#[cfg(test)]
struct TestAgent {
    agent: AgentCore,
    // ...
}

#[cfg(test)]
impl TestAgent {
    async fn new_with_response(response: LlmResponse) -> Self { ... }
```

## 5.3 [建议] `custom.rs` 的 `tokenize_argv` 错误消息重映射可以简化

**优化前** (`custom.rs:57-70`, 14 行):
```rust
fn tokenize_argv(input: &str) -> Result<Vec<String>> {
    reject_unquoted_shell_metacharacters(input)?;
    tokenize_quoted_args(input).map_err(|error| match error.to_string().as_str() {
        "unclosed quote in command / 命令中有未闭合的引号" => {
            anyhow::anyhow!("unclosed quote in custom tool command / 自定义工具命令中有未闭合的引号")
        }
        "empty command / 空命令" => {
            anyhow::anyhow!("custom tool command is empty / 自定义工具命令为空")
        }
        _ => error,
    })
}
```

**问题:** 通过字符串内容匹配错误消息来决定重映射，极其脆弱——如果上游消息改一个字就静默失效。

**优化后** (使用 `context()` 包装而不是替换):
```rust
fn tokenize_argv(input: &str) -> Result<Vec<String>> {
    reject_unquoted_shell_metacharacters(input)?;
    tokenize_quoted_args(input)
        .context("custom tool command tokenization failed / 自定义工具命令解析失败")
}
```
或者让 `tokenize_quoted_args` 返回类型化错误枚举，而非依赖字符串匹配。
